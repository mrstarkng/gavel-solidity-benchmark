# Source Snapshot Publication Plan

This plan describes how to publish Solidity source files for reproducibility without committing them to `main` yet.

## Recommendation

Keep `main` lightweight and publish an optional GitHub Release artifact containing a minimal manifest-referenced source snapshot.

Recommended initial snapshot:

- snapshot type: minimal manifest-referenced source snapshot;
- contents: only Solidity files referenced by `data/manifest/gavel_solidity_benchmark_manifest.jsonl`;
- layout: preserve project slug and original relative source path;
- publication channel: GitHub Release artifact, not committed into `main`;
- license stance: benchmark metadata/scripts remain MIT, Solidity files remain under upstream licenses.

Do not publish the full 30,816-file local source tree unless a later review explicitly approves that higher-risk option.

## Why Not Commit Sources to `main`

Committing source files into `main` can make it look as if this repository maintains or relicenses the upstream project code. That is not the intent.

The current MIT license should continue to apply to:

- manifests;
- metadata;
- documentation;
- scripts;
- source lock/provenance files generated by this project.

It should not be presented as relicensing third-party Solidity files from Code4rena or upstream projects.

## Proposed Release Artifact Layout

Package name:

```text
gavel-solidity-source-snapshot-minimal-<version>.tar.gz
```

Suggested internal layout:

```text
README.source_snapshot.md
SOURCE_PROVENANCE.json
THIRD_PARTY_LICENSES.md
source_snapshot_sha256sums.txt
sources/
  <project_slug>/
    <original/source/path.sol>
```

The `sources/` directory should contain only the manifest-referenced source files.

## Required Files

### `README.source_snapshot.md`

Must state:

- this is an optional source snapshot for GAVEL Solidity Benchmark reproducibility;
- files are third-party Solidity source files copied from Code4rena/project repositories;
- benchmark metadata/scripts are MIT, but Solidity files remain under upstream licenses;
- the snapshot is scoped to manifest-referenced files, not full project source distributions;
- users should consult upstream repositories for complete projects and license terms.

### `SOURCE_PROVENANCE.json`

Must include one entry per copied source file:

- `project_slug`
- `source_path`
- `repo_url`
- `commit_sha` or `commit_ref` if available
- `sha256`
- `size_bytes`
- `license_status`
- source lockfile version/reference

The candidate file `data/sources/source_snapshot_candidate_manifest.json` is a local planning input for this.

### `THIRD_PARTY_LICENSES.md`

Must include:

- a top-level notice that Solidity files are not relicensed under this repository's MIT license;
- per-project rows for `project_slug`, `repo_url`, `commit_sha`/`commit_ref`, and `license_status`;
- explicit `needs review` markers for projects where license status is unknown;
- guidance to inspect upstream repositories before redistribution or commercial use.

### `source_snapshot_sha256sums.txt`

Must include SHA256 checksums for:

- every copied `.sol` file;
- `README.source_snapshot.md`;
- `SOURCE_PROVENANCE.json`;
- `THIRD_PARTY_LICENSES.md`.

## Candidate Build Procedure

After approval, build outside tracked source paths, for example:

```bash
mkdir -p /private/tmp/gavel_source_snapshot_minimal/sources
python3 scripts/validate_manifest.py
python3 scripts/validate_sources.py --sources-root local_workspace/sources/solidity_sources
python3 -m json.tool data/sources/source_snapshot_candidate_manifest.json
```

Then copy only the files listed in `data/sources/source_snapshot_candidate_manifest.json` into:

```text
/private/tmp/gavel_source_snapshot_minimal/sources/<project_slug>/<source_path>
```

Do not create `data/source_snapshot/` or commit `.sol` files unless a later decision explicitly changes the policy.

## Option Comparison

| Option | Publish channel | Scope | Reproducibility | License/provenance risk | Recommendation |
|---|---|---|---|---|---|
| No source snapshot | `main` only | Manifest + lockfile + scripts | Good if upstream repos remain available | Low | Acceptable baseline |
| Minimal source snapshot | Release artifact | 312 referenced `.sol` files | Strong for benchmark row validation | Medium | Preferred |
| Full local source snapshot | Release artifact only, if ever | 30,816 `.sol` files plus broader context | Stronger offline context, not necessarily needed for scoring | High | Defer |
| Commit source snapshot to `main` | Git history | Minimal or full | Convenient but heavy | Medium to high | Not recommended |

## License and Provenance Controls

Controls required for any source snapshot:

- preserve original file contents and license headers;
- do not strip comments or headers;
- do not apply repository MIT license to Solidity source files;
- include `THIRD_PARTY_LICENSES.md`;
- include file-level `SOURCE_PROVENANCE.json`;
- include checksums;
- label unknown license projects as requiring review;
- keep source snapshot outside `main` unless explicitly approved later.

## Decision Gate

Recommended lead decision:

1. Approve the minimal manifest-referenced snapshot as a release artifact candidate.
2. Review license/provenance status for the 81 projects, especially the 42 projects with unknown license status in the source lockfile.
3. Generate the release artifact outside tracked paths.
4. Attach it to a GitHub Release only after review.

Until that decision is made, the repository should continue to publish only manifest data, source lock/provenance metadata, and fetch/validate scripts.
